About this tool

A side-by-side comparison of Microsoft Enterprise Agreement (EA) pricing against Microsoft Customer Agreement — Enterprise (MCA-E) retail pricing for Azure services. Designed to help sellers, finance teams, and customers understand the cost impact of moving from EA to MCA-E.

What it does

Looks up live Azure Retail prices and compares them with EA pricing you supply (either by entering a SKU or uploading a usage export). Handles graduated tiers, Reservations (1Y/3Y, upfront and monthly), Savings Plans, and Spot.

What it doesn't do

It does not access your Azure account, store any data, or apply your specific commercial discounts unless you enter them (ACD %). All processing happens in your browser.

Data sources

Pricing comes from the public Azure Retail Prices API (prices.azure.com). EA prices come from your own input/files. No third-party services.

Privacy

The app runs locally. Files you upload are parsed in the browser and never sent anywhere. The only outbound call is to the Microsoft pricing API.

Audience

  • Sellers / CSAs — quickly model EA-to-MCA-E migrations
  • Finance / FinOps — understand cost impact of agreement changes
  • Customers — sanity-check MCA-E quotes against current EA rates

How to use

Three modes for three workflows — from a quick single lookup to a full usage-file analysis.

Pick your mode

Manual Search

Best for: quickly checking a single product or SKU. Type a product/SKU name (e.g. "Premium SSD P10"), optional region, hit Search.

Bulk Lookup

Best for: pricing a list of known products. Paste up to a few hundred names, one per line. Each is looked up against the Retail API.

Usage File Analysis

Best for: full EA-vs-MCA comparison. Upload a CSV/XLSX export of EA usage. EA prices come from the file; MCA-E prices are looked up live.

Step-by-step (Usage File Analysis)

1

Set the global controls

Pick your Country → this auto-selects Pricing Currency and Billing Currency. Set ACD % if you have an Azure Consumption Discount on MCA-E (e.g. 10 for 10% off).

2

Upload your usage file

Go to Usage File Analysis → pick your CSV/XLSX. The tool reads MeterId, Cost, Quantity, UnitPrice, PricingModel, ChargeType, Term, Frequency, etc.

3

Click Analyze File

Rows are grouped by MeterId + PricingModel + Term + ChargeType + Frequency so OnDemand, Reservation, Savings Plan and Spot rows stay separate. The Retail API is then called for each unique meter.

4

Read the output

For each meter you'll see: EA vs MCA-E unit price (with per-tier breakdown for graduated SKUs), totals, Comparable Delta % based on totals (not unit prices), and the row Type (e.g. Reservation 3Y (Purchase · Monthly)).

5

Verify and export

Click Verify API on any row to open the raw Retail API response in a new tab. Click Download CSV to export the comparison.

Reading the comparison

Type column

  • OnDemand — standard pay-as-you-go
  • Reservation 3Y (Purchase · Upfront) — one-time RI payment compared to MCA upfront price
  • Reservation 3Y (Purchase · Monthly) — monthly RI billing compared to MCA upfront ÷ 36
  • * suffix means the frequency was auto-detected (Frequency column was missing)
  • SavingsPlan 1Y / 3Y — per-hour rate from the savings plan API field
  • Spot — matched against Spot SKUs only

Comparable Delta %

For graduated/tiered SKUs, comparing unit prices is misleading because different quantities consume different tiers. The delta is therefore computed on EA Total Cost vs MCA-E Total Cost, which is the apples-to-apples figure.

$0 RI consumption rows

If your file is an Actual export (not Amortized), Reservation Usage rows show Cost = 0 because the RI absorbed the cost. These rows are suppressed automatically — the Purchase rows carry the real cost.

How I built this app

Architecture at a glance

Local desktop app

A single self-contained report.html file plus a tiny Python launcher (launcher.py) that serves the HTML on localhost and proxies the Azure Retail Prices API to bypass browser CORS. Everything is bundled into a single .exe with PyInstaller and wrapped in an Inno Setup installer.

Web version (Azure SWA)

The same HTML hosted on Azure Static Web Apps, with the Python proxy reimplemented as a Node 18 Azure Function (/api/prices). CI/CD via GitHub Actions; ~40 lines of Node, no dependencies.

Key design decisions

Single-file SPA, no build step

Whole app is one report.html file with HTML/CSS/JS inline. No bundler, no framework, no node_modules. Trade-off: no component reuse, but easy to deploy, easy to read, zero supply-chain risk.

Why a CORS proxy was needed

The Azure Retail Prices endpoint doesn't return CORS headers, so browsers block direct calls. A tiny passthrough on the same origin solves this — the browser sees a same-origin request and the server-to-server hop to prices.azure.com isn't subject to CORS. Locally that's Python; on SWA it's a Node Function.

Browser-side parsing

Usage files (CSV/XLSX) are parsed entirely client-side — CSVs streamed in chunks via the File API, XLSX via SheetJS. No file ever leaves the browser. Chosen for privacy and to keep server cost at zero.

Grouping that respects pricing model

Lumping all rows for a single meter together produced wildly wrong results (mixing OnDemand with Reservation, monthly with upfront, etc.). The grouping key is now MeterId + PricingModel + Term + ChargeType + Frequency so each economic model is compared like-for-like.

Tiered/graduated pricing

Many Azure meters have tiers (e.g. first 5 TB at one rate, next 50 TB cheaper, etc.). Showing a "blended unit price" was misleading, so the MCA-E unit cell renders each tier rate stacked in a single row, and the Comparable Delta % uses totals not unit prices.

Reservation upfront vs monthly

The Retail API only returns RI prices as full upfront amounts. Customers paying monthly see one month's installment in their EA file, which produced ~3,500% nonsense deltas before. The fix: read the Frequency column (OneTime vs Recurring); if missing, auto-detect by comparing the EA cost-per-RI to upfront and upfront / months, picking the closer match. Monthly comparison divides MCA upfront by months in the term.

Currency handling

Country selection auto-derives Pricing and Billing currencies (using Microsoft's MCA FAQ table verbatim). The Retail API is queried in the chosen pricing currency; no FX maths in the app.

Sticky legal disclaimer

A permanent (non-dismissable) Fluent MessageBar at the bottom of every screen. Required because the tool produces non-authoritative pricing estimates and is not endorsed by Microsoft.

How I actually built this

The whole thing was built iteratively in collaboration with an AI coding assistant. I started with a one-off question — "can you look up these EA part numbers in the Azure Retail Prices API and tell me the MCA equivalent?" — and the project grew from there as real edge cases surfaced.

The workflow was: describe what I wanted in plain English, get a working draft, run it against real EA usage files, find something wrong, describe the problem, repeat. Each round added one capability or fixed one bug. There was no upfront design document — the architecture emerged from the conversation.

A few specific milestones along the way:

  • Started as one HTML file with a hard-coded list of products. Quickly hit the browser CORS wall on the Retail Prices API, so I added a tiny Python launcher to act as a same-origin proxy.
  • Tiered pricing was the first big surprise — comparing a blended unit price gave misleading deltas, so the math moved to per-tier rates and total-cost comparison.
  • Pricing model awareness came after a D4 v3 row showed an EA total that didn't equal qty × unit price. Turned out OnDemand and Reservation rows were being merged. The group key now includes PricingModel + Term + ChargeType.
  • Reservation upfront vs monthly was the most subtle bug — the Retail API only returns full-upfront RI prices, but customers paying monthly were showing 3,500% deltas. Solved by reading the Frequency column (with an auto-detect fallback when it's missing).
  • Packaging happened late — once the app was useful enough to share, I wrapped it with PyInstaller and Inno Setup for Windows, then mirrored it to Azure Static Web Apps for a shareable URL.

Total elapsed time was a handful of sessions over a couple of weeks. The code base is intentionally small (one HTML file, ~1,500 lines) because every line still has to make sense to one person.
