About this tool

A side-by-side comparison of Microsoft Enterprise Agreement (EA) pricing against Microsoft Customer Agreement — Enterprise (MCA-E) retail pricing for Azure services. Designed to help sellers, finance teams, and customers model the cost impact of moving from EA to MCA-E — from a single SKU to a full month's invoice.

Why this exists

With the EA-to-MCA-E transition becoming mandatory at renewal, sellers and customers need to model the true cost impact — not just the headline rates. For customers benefiting from EA price protection (notably in Japan, where MCA-E reintroduces local FX exposure), the delta can be material.

Until now this has been a highly manual, error-prone, and unscalable exercise. Spot-checking the top 10 SKUs misses the bulk of spend; comparing a full month's invoice in Excel runs straight into the fact that EA and MCA-E use different naming conventions, different units of measure, and different region labels for the same underlying meter.

Why naïve matching fails

Name drift

EA product names are not identical to MCA MeterId product names. Abbreviations, casing, and word order all differ.

Unit of Measure

EA SKUs and MCA MeterIds frequently use different UoM strings (e.g. 100 Hours vs 1 Hour, 1/Month vs 1 Month).

Region mapping

EA region labels do not always match MCA region labels for the same physical region.

Non-unique MeterIds

The same MeterId can be shared across regions, or split across regions, depending on the service. Both cases break a simple lookup.

Graduated tiers

MCA has tier-minimum-unit pricing (e.g. first 5 TB at one rate, next 50 TB cheaper). Comparing a blended unit price gives misleading deltas.

Pricing models

OnDemand, Reservations (1Y/3Y, Upfront/Monthly), Savings Plans and Spot all need to be compared like-for-like, not lumped together.

What you get

Per-meter EA vs MCA-E comparison

For every meter in your input: EA customer unit price, MCA-E retail price (with ACD applied if set), per-tier breakdown for graduated SKUs, totals, and a Comparable Delta % computed on totals (not unit prices).

Full-invoice scale

Handles full EA usage exports — thousands of rows — not just a handful of SKUs. Rows are grouped by meter + pricing model so each economic model stays separate.

Automatic name resolution

You don't need to massage your input. The tool normalizes EA names, resolves part numbers directly to MeterIds, and falls back to a catalog-driven fuzzy matcher when names diverge.

Drill-down and export

Verify any row against the raw Azure Retail API response, expand the results table to full screen, and download the full comparison as CSV.

What it does not do

  • It does not access your Azure account or tenant.
  • It does not store, transmit, or persist your usage files — parsing happens entirely in the browser.
  • It does not apply your specific commercial discounts unless you enter them (ACD %).
  • It does not produce an authoritative quote — always verify against official Microsoft pricing documents.

Audience

  • Sellers / CSAs — quickly model EA-to-MCA-E migrations for renewal conversations
  • Finance / FinOps — understand cost impact of agreement changes and FX exposure
  • Customers — sanity-check MCA-E quotes against current EA rates

Data sources & privacy

Pricing: public Azure Retail Prices API (prices.azure.com). EA prices: from your own input or uploaded file. No third-party services.

The app runs locally in your browser. The only outbound network call is to the Microsoft pricing API. Uploaded files never leave your machine.

How to use

Two modes for two workflows — from a quick lookup of a few known SKUs to a full usage-file invoice comparison.

Pick your mode

Bulk Lookup

Best for: pricing a list of known products or part numbers. Paste up to a few hundred entries, one per line. Each is resolved against the Azure Retail Prices API.

Usage File Analysis

Best for: full EA vs MCA-E invoice comparison. Upload a CSV / XLSX / JSON export of EA usage. EA customer unit prices come from the file; MCA-E retail prices are looked up live.

Before you start — header controls

These apply to both modes:

  • Country — auto-selects Pricing Currency and Billing Currency using Microsoft's MCA FAQ table.
  • ACD % — your Azure Consumption Discount on MCA-E. Enter 10 for 10% off. Leave blank for list price.
  • Theme toggle — switch between light and dark (Fluent tokens).

When names don't match

You don't need to massage your input. The tool resolves each entry in this order:

  1. Part-number index — if the input is a known EA part number, jump straight to the matching MeterId.
  2. Normalized name — strip unit-of-measure noise, expand EA aliases, remap region names, then try exact match.
  3. Catalog matcher — abbreviation-scored fuzzy match against the offline Azure product catalog (the same catalog the portal uses).
  4. Unresolved — if nothing matches, the row is flagged so you can correct it manually.

In practice this resolves >95% of rows from a real EA usage export without intervention.

Reading the comparison

Type column

  • OnDemand — standard pay-as-you-go
  • Reservation 3Y (Purchase · Upfront) — one-time RI payment compared to MCA upfront price
  • Reservation 3Y (Purchase · Monthly) — monthly RI billing compared to MCA upfront ÷ 36
  • * suffix means the frequency was auto-detected (Frequency column was missing)
  • SavingsPlan 1Y / 3Y — per-hour rate from the savings plan API field
  • Spot — matched against Spot SKUs only

Comparable Delta %

For graduated/tiered SKUs, comparing unit prices is misleading because different quantities consume different tiers. The delta is therefore computed on EA Total Cost vs MCA-E Total Cost, which is the apples-to-apples figure.

$0 RI consumption rows

If your file is an Actual export (not Amortized), Reservation Usage rows show Cost = 0 because the RI absorbed the cost. These rows are suppressed automatically — the Purchase rows carry the real cost.

Architecture at a glance

Web version (Azure SWA)

A single self-contained HTML file hosted on Azure Static Web Apps, with the Azure Retail Prices API proxied through a Node 18 Azure Function (/api/prices) to bypass browser CORS. Deployed from the Azure DevOps garage-static-pages repo with separate DEV and PROD branches. ~40 lines of Node, no dependencies.

The matching logic (the heart of the tool)

The whole reason this tool exists is that you cannot match an EA usage row to an MCA-E meter with a naive string compare. Every match goes through this exact pipeline:

Step 1 — Part-number index (PART_INDEX)

At startup, loadPartIndex() builds a hash from a pre-generated part-index.json mapping each known EA part number directly to its MCA MeterId. If the input is a part number, this is a single O(1) lookup and we are done — no name matching needed.

Step 2 — EA SKU normalization (normalizeEaSku)

If the input is a name, it goes through three normalizations before any lookup:

  • UOM_STRIP_RE — a regex that removes unit-of-measure noise from product names: 100 Hours, 1/Month, 10K Transactions, etc. EA and MCA disagree on these strings constantly.
  • EA_ALIASES — a static dictionary that expands EA shorthand to MCA canonical names (e.g. VMs → Virtual Machines, Mgd Disks → Managed Disks).
  • EA_REGION_MAP — remaps EA region labels (e.g. AP East) to the Azure ARM region slugs the Retail API actually accepts (eastasia).

After normalization, an exact match against the catalog is attempted. If it hits, we're done.

Step 3 — Catalog matcher (catalogMatch + AZ_CATALOG)

If exact match fails, loadCatalog() serves a pre-built offline copy of the Azure product catalog (az-catalog.json) — the same source of truth the Azure portal uses for naming. The matcher scores every candidate using abbrevScore(), which weights token overlap and abbreviation expansion (e.g. P10 matches Premium SSD P10; D4v3 matches D4 v3). The highest-scoring candidate above a threshold wins.

This is what took the real-world hit rate on EA usage files from about 22% (naive normalized match) to about 96%.

Step 4 — Unresolved

If nothing scores above threshold, the row is flagged Unresolved in the output rather than silently failing. The user sees exactly which inputs need manual attention.

Resolution order at a glance: PART_INDEX → normalizeEaSku + exact match → catalogMatch (fuzzy + abbreviation-scored) → Unresolved.

Other key design decisions

Single-file SPA, no build step

Whole app is one report.html file with HTML/CSS/JS inline. No bundler, no framework, no node_modules. Trade-off: no component reuse, but easy to deploy, easy to read, zero supply-chain risk.

Why a CORS proxy was needed

The Azure Retail Prices endpoint doesn't return CORS headers, so browsers block direct calls. A tiny passthrough on the same origin solves this — the browser sees a same-origin request and the server-to-server hop to prices.azure.com isn't subject to CORS. Locally that's Python; on SWA it's a Node Function.

Browser-side parsing (CSV / XLSX / JSON)

Usage files are parsed entirely client-side — CSVs streamed in chunks via the File API, XLSX via SheetJS, JSON natively. No file ever leaves the browser. Chosen for privacy and to keep server cost at zero.

Grouping that respects pricing model

Lumping all rows for a single meter together produced wildly wrong results (mixing OnDemand with Reservation, monthly with upfront, etc.). The grouping key is now MeterId + PricingModel + Term + ChargeType + Frequency so each economic model is compared like-for-like.

Tiered/graduated pricing

Many Azure meters have tiers (e.g. first 5 TB at one rate, next 50 TB cheaper, etc.). Showing a "blended unit price" was misleading, so the MCA-E unit cell renders each tier rate stacked in a single row, and the Comparable Delta % uses totals not unit prices.

Reservation upfront vs monthly

The Retail API only returns RI prices as full upfront amounts. Customers paying monthly see one month's installment in their EA file, which produced ~3,500% nonsense deltas before. The fix: read the Frequency column (OneTime vs Recurring); if missing, auto-detect by comparing the EA cost-per-RI to upfront and upfront / months, picking the closer match. Monthly comparison divides MCA upfront by months in the term.

Currency handling

Country selection auto-derives Pricing and Billing currencies (using Microsoft's MCA FAQ table verbatim). The Retail API is queried in the chosen pricing currency; no FX maths in the app.

UI niceties

Fluent-token theme with light/dark toggle, sticky top controls, expand-to-full-screen on any results table, and a permanent (non-dismissable) Fluent MessageBar at the bottom of every screen carrying the legal disclaimer.

How I actually built this

This tool exists because the problem it solves is genuinely hard, and almost no one was looking at it the right way. The naïve approaches — "just match on product name", "just compare the top 10 SKUs", "just average the unit prices" — all sound reasonable for ten minutes and then fall apart on the first real customer invoice. Getting it right meant repeatedly recognising that what looked like a bug was actually a category of problem nobody had named yet.

Almost every milestone below started the same way: a real EA usage file produced an answer that was obviously wrong, and figuring out why it was wrong was the actual work. The code was the easy part. The hard part was framing the problem precisely enough that it had a clean solution.

How the tool matches an EA row to an MCA-E meter

Every input goes through the same four-stage pipeline. Each stage is tried in order; the first one that succeeds wins.

  1. Direct part-number lookup. If the input is already an EA part number, a prebuilt lookup table maps it straight to the matching MCA meter. No name matching is needed, and the answer is exact.
  2. Cleaned-up name match. If the input is a product name, it's first put through three clean-ups: unit-of-measure noise is stripped out, EA shorthand is expanded to the full Microsoft product name, and EA region labels are translated to the official Azure region names. The cleaned-up name is then compared to the catalog for an exact hit.
  3. Fuzzy catalog match. If there's still no exact hit, the name is scored against every entry in an offline copy of the official Azure product catalog — the same catalog the Azure portal itself uses. The scoring is abbreviation-aware, so a short form like "D4 v3" still matches "Virtual Machines D4 v3" and "P10" still matches "Premium SSD Managed Disks P10". The highest-scoring candidate above a confidence threshold wins.
  4. Flag as unresolved. If nothing scores high enough, the row is marked Unresolved in the output rather than guessed. The user sees exactly which inputs need a manual look — a wrong answer is worse than no answer when the customer is making a multi-year commercial decision.

On real EA usage files, this pipeline resolves around 19 in 20 rows without any manual intervention.

  • Tiered pricing. The first usage file came back with deltas that felt off. The instinct to compare unit price to unit price is so natural that it took a deliberate step back to realise that for any meter with graduated tiers, unit prices are mathematically not comparable — only totals are. That reframing changed how every downstream calculation in the tool works.
  • Pricing model awareness. A virtual machine row showed an EA total that didn't equal quantity times unit price. The easy reaction would have been to call it a data quality issue. The actual cause was deeper: pay-as-you-go, reservation, savings plan, and spot rows for the same meter were being silently combined. Recognising that one meter ID can represent four entirely different economic models — and that the grouping key therefore has to include pricing model, term, charge type, and frequency — was the design call that made the comparison trustworthy.
  • Reservations paid monthly. When deltas of 3,500% started appearing, it would have been tempting to dismiss the affected rows. Tracing through the math instead surfaced something fundamental: the Retail API only quotes reservations as a full upfront amount, but customers paying monthly only see one twelfth or one thirty-sixth of that in any given file. The fix — read the billing frequency from the file, and when it's missing, infer it by checking which interpretation of the EA cost lines up with the API price — is the kind of thing you only design once you've decided the tool has to work on real files, not idealised ones.
  • Why naïve name matching fails. First-pass matching hit about one row in five. Diagnosing why took looking at hundreds of mismatched pairs and noticing three distinct patterns hiding inside what looked like one problem: units of measure differ between EA and MCA, EA uses its own abbreviations, and EA region labels are not the same strings as Azure ARM region slugs. Splitting one vague "name matching is hard" into three concrete, separately solvable problems was the breakthrough — each one then got its own small, targeted fix.
  • Part numbers. Once name matching was working, the next observation was that a meaningful share of customer inputs are already EA part numbers, not names — and a part number is a primary key. Designing the resolver as a pipeline (part number first, normalised name second, fuzzy match third) instead of one monolithic matcher meant the common case became a single hash lookup, and the harder logic only runs when it's actually needed.
  • The catalog matcher. Even after the cleanups, the strict match was still missing roughly a third of rows, and the failures were not random — they were abbreviated SKU names that any human reading them would recognise immediately. That observation led to the key design decision: stop comparing against strings, start comparing against the official Azure product catalog itself, and score candidates by how well their tokens line up rather than whether they match character for character. Real-world hit rate jumped from around one in five rows to roughly nineteen in twenty.
  • Knowing when not to fail silently. A subtle but important call: when nothing matches with sufficient confidence, the row is flagged Unresolved rather than guessed. A wrong answer is worse than no answer when the customer is making a multi-year commercial decision.
  • Hosting. Delivered as a hosted web app on Azure Static Web Apps rather than a local install — zero footprint for sellers and finance teams, and instant updates. The whole tool is one self-contained HTML file, deliberately kept that way to avoid divergence.

The code base is intentionally small — one HTML file, around 2,700 lines — because the value is in the decisions encoded into it, not the line count. Every rule in the matcher, every grouping key, every special case for reservations exists because a real customer invoice forced it into existence.
